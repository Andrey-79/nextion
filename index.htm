Справочная по программированию дисплеев Nextion на русском.

№           Общие правила

1             Все инструкции по последовательному интерфейсу: заканчиваются тремя байтами 0xFF 0xFF 0xFF, то есть: десятичные: 255 или шестнадцатеричные: 0xFF, или ansichar: ÿ или двоичные: 11111111,

                т.е. byte ndt[3] = {255,255,255}; write(NDT, 3); или print(«\ xFF \ xFF \ xFF»); или print(«ÿÿÿ»)

2             Все инструкции и параметры в ASCII

3             Все инструкции строчными буквами

4             Блоки кода заключенные в фигурные скобки {} не могут быть отправлены через последовательный интерфейс, 

                это означает, что команды нельзя передавать через последовательный интерфейс

5             Пробел 0x20 используется для отделения команды от параметров.

6             В параметрах нет пробелов, если не указано иное

7             Nextion использует целочисленную математику и не имеет блока плавающей поддержки.

8             Присвоение несложно оценивается полностью при достижении значения после оператора.

9             Оценка сравнения не сложна, но может быть объединена (&& и ||).

10           Инструкции порта обрабатываются при получении завершения (см. 1.1 )

11           Экраные символовы выполняется с использованием таких значений: \ r  0x0D, \”   0x22 , \\   0x5C

12           Nextion не поддерживает порядок выполнения операций. sys0 = 3 + (8 * 4) не работает.

13           16-битные 565 цветов в десятичном виде от 0 до 65535 (см. Примечание )

14           Текстовые значения должны быть заключены в двойные кавычки: т.е. «Hello»

15           Есть элементы специфичные для усовершенствованных моделей, отмечены отдельно

16           Режим передачи данных (используется командами addt и wept )

                MCU отправка в Nextion

                MCU отправляет команду. то есть: wept 30,20ÿÿÿ или addt 1,0,320ÿÿÿ

                Nextion требуется ~ 5 мс для подготовки к передачи данных

                Nextion отправляет «Ready» 0xFE 0xFF 0xFF 0xFF Возвращаемые данные (см. 7.32 )

                MCU теперь может отправлять указанное количество (20) необработанных байтов в Nextion

                Nextion получает необработанные байты от MCU, пока не будет получено указанное количество (20)

                Nextion отправляет «Finished» 0xFD 0xFF 0xFF 0xFF Возвращаемые данные (см. 7.31 )

                MCU и Nextion могут перейти к следующей команде

                Примечание. На шаге 5 связь будет оставаться в ожидании до тех пор, пока не будет получен каждый байт указанного количества. 

                - В течение этого времени Nextion не может выполнять какие-либо другие команды и может действительно зависать, если MCU не может доставить количество байтов, как указано в параметре команды. 

                - количество данных ограничено последовательным буфером (все команды + окончания + данные <1024)

17           Только атрибуты компонента в зеленых и не только для чтения системных переменных могут быть назначены новые значения во время выполнения. Все остальные только для чтения во время выполнения, за исключением .objname

18           Числовые значения можно вводить с HEX байтом выравнивания  т.е.: n0.val = 0x01FF

19           Дополнительно . Адресный режим - это расширенный метод, добавляющий последовательную инструкцию с двумя байтами для адреса. Двухбайтовый адрес должен быть отправлен в порядке с прямым порядком байтов, то есть:

                2556 отправляется 0xFC 0x09. По умолчанию адрес Nextion равен 0 и не требует двухбайтового префикса. Когда используется двухбайтовая адресация, Nextion будет отвечать на команду только в том случае,

                если ее адрес совпадает с двухбайтовым префиксом или если переданный адрес - 65535 широковещательных сообщений. Смотрите системную переменную addr.

20           Дополнительно . Режим повторного анализа протокола - это продвинутый метод, который позволяет пользователям определять свой собственный входящий протокол и обработку поступающих последовательных данных. 

                В активном режиме повторной обработки протокола входящие последовательные данные не будут обрабатываться встроенным ПО Nextion, а будут ожидать обработки в последовательном буфере. Для выхода из активного режима

                 повторной обработки протокола recmod должен быть установлен обратно в пассивный (то есть: в логике Nextion как recmod = 0), чего нельзя достичь через последовательный порт. 

                Отправьте DRAKJHSUYDGBNCJHGJKSHBDNÿÿÿ через последовательный порт для последовательного выхода из активного режима.Большинство приложений HMI не требуют режима

                повторной обработки протокола и должны быть пропущены, если не полностью поняты.

21           Комментирование пользовательского кода внутри Events использует технику двойной косой черты (два символа прямой косой черты /). См. 2.30 для правильного использования.
